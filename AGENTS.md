# AGENTS.md

agent 交互时，使用中文回复。

## 工作流编排

### 1. 默认先规划

- 对任何非平凡任务（3+ 步骤或涉及架构决策）都要先进入规划模式
- 如果出现问题，立即停止并重新规划 — 不要硬推
- 验证步骤也要用规划模式，不只是构建阶段
- 提前写详细的 spec 以减少歧义

### 2. 子代理策略

- 大量使用子代理（subagent）保持主上下文窗口清洁
- 将研究、探索、并行分析任务卸载给子代理
- 对复杂问题，通过子代理投入更多算力
- 每个子代理一个专注任务

### 3. 自我改进循环

- 每次用户纠正后：更新 `tasks/lessons.md` 记录模式
- 为自己写规则，防止同样的错误
- 持续迭代这些经验教训，直到错误率下降
- 会话开始时回顾相关项目的经验教训

### 4. 完成前必须验证

- 永远不要在没有证明可行的情况下标记任务完成
- 在相关时对比 main 分支和你的改动的行为差异
- 问自己："资深工程师会批准这个吗？"
- 运行测试、检查日志、证明正确性

### 5. 追求优雅（平衡）

- 对非平凡改动：暂停并问"有没有更优雅的方式？"
- 如果修复感觉很 hack：考虑"基于我现在掌握的所有信息，实现优雅的解决方案"
- 对简单明显的修复跳过这一步 — 不要过度工程化
- 在呈现之前挑战自己的工作

### 6. 自主修复 Bug

- 收到 bug 报告时：直接修复，不要要求手把手指导
- 指出日志、错误、失败的测试 — 然后解决它们
- 用户零上下文切换
- 主动修复失败的 CI 测试，不需要被告知如何做

## 任务管理

1. **先规划**：将计划写入 `tasks/todo.md`，包含可检查的条目
2. **验证计划**：开始实现前先确认
3. **跟踪进度**：边做边标记完成的条目
4. **解释变更**：每一步给出高层次总结
5. **记录结果**：在 `tasks/todo.md` 添加回顾部分
6. **捕获经验**：被纠正后更新 `tasks/lessons.md`

## 发布相关

- TestFlight 发布时需要 `MATCH_PASSWORD` 环境变量，定义在项目根目录 `.env` 文件中
- 执行 testflight.sh 前需先 `source .env` 或 `export $(cat .env | xargs)`

## 核心原则

> 以下准则偏向谨慎而非速度。对平凡任务可自行判断跳过。

### 1. 先想再写

**不要假设，不要隐藏困惑，主动暴露权衡。**

动手实现之前：

- 明确说出你的假设。不确定就问。
- 如果存在多种解读，全部列出 — 不要默默选一个。
- 如果有更简单的方案，说出来。该 push back 就 push back。
- 如果有不清楚的地方，停下来，指出哪里不清楚，然后问。

### 2. 简单优先

**只写解决问题所需的最少代码，不做投机性设计。**

- 不做没被要求的功能。
- 不为只用一次的代码做抽象。
- 不做没被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 如果写了 200 行但 50 行就能搞定，重写。

自检："资深工程师会说这太复杂了吗？" 如果是，简化。

### 3. 精准改动

**只动必须动的，只清理自己造成的。**

编辑已有代码时：

- 不要"顺手改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你会用不同的方式写。
- 发现不相关的死代码，提一句 — 不要删。

当你的改动产生孤儿代码时：

- 删除因你的改动而变得无用的 import/变量/函数。
- 不要删除改动前就已存在的死代码，除非被要求。

检验标准：每一行改动都应能直接追溯到用户的请求。

### 4. 目标驱动执行

**定义成功标准，循环直到验证通过。**

把任务转化为可验证的目标：

- "加验证" → "为非法输入写测试，然后让测试通过"
- "修 bug" → "写一个复现测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步任务先列简要计划：

```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

强成功标准让你能独立循环推进；弱标准（"让它能跑"）需要反复确认。

### 5. 不偷懒

找到根本原因，不做临时修复，保持资深开发者标准。
